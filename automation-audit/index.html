3 · Trigger-based (not cron)

Replicate the research-os GitHub Actions pair

@claude mentions + automatic Claude PR review exist in exactly one repo. Copy claude.yml + claude-code-review.yml to tx-dash, call-infra, monitor, the-tool — the four repos with real deploy surface.

Post-deploy verify on prod deploys

tx-dash staging is the only auto-verified deploy. Add a workflow_run job after prod deploys that runs the e2e smoke and pings Slack on failure — the cloud equivalent of /tx-e2e-verify and /altis-smoketest.

Pre-Loop health gate

monitor TODO wants goldens + anchor capture before each 5:17 run. Cheapest as a step inside the Cloud Run Job itself, not a separate routine — flagging so it doesn't get cron'd unnecessarily.

One-off reminders via run_once_at

Hiring take-home walkthroughs (24h-before prep), quarterly acquired-entity re-checks, "verify X by date" chores — cheap disposable routines instead of calendar todos.

4 · Constraints to plan around

  • 1-hour minimum interval on cloud routines — sub-hourly sweeps (stuck-asset recovery at 30-min) belong in Cloud Scheduler/pg_cron like the existing sweepers; Claude routines are for the judgment layer on top.
  • Connected MCP connectors are Slack and Ramp only. Routines needing Supabase must go through repo-based access (SQL over API with keys provisioned in the altis-crons environment) or you connect the Supabase MCP connector first.
  • Cloud agents can't see the laptop. Anything migrating off launchd must get its inputs from git, APIs, or MCP — the kg-reconcile pattern (fresh clone of main, prompt = skill name) is the proven template.
  • Crons are UTC — the ET times above shift by an hour when DST ends in November.
  • Keep the write-approval doctrine: report-and-propose routines (janitor, DQ sweeps) should stay read-only; destructive fixes go through PRs or human approval, consistent with the existing Supabase write rule.
