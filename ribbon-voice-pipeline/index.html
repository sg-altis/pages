System Design — Voice Prescreen Enrichment

The Voice Enrichment Loop

A CRM row goes out, an AI voice interviewer calls the person, and the row comes back carrying what they claimed and how much to believe it. Four systems, one join key, no human in the middle until a real expert surfaces.

The loop closes because a single value — crm_id — rides every hop: into the dynamic variables that personalize the call, back out on the conversation record, through the post-call webhook, onto the HubSpot write. Get the join key right and enrichment is deterministic. Get it wrong and you have orphaned transcripts.

01

The four systems

HubSpot flat · Supabase holds the mess

Each does one job. The rule that keeps it sane: transcripts, nested analysis, webhook idempotency, and per-contact call state live in Supabase. HubSpot only ever sees clean, final columns.

HubSpot
record
Source of the contact list (email, phone, persona, target company) and destination for the enriched columns.
Supabase
spine
Link + variable generation, the call state machine, webhook receiver, transcript store, the parser.
Lemlist
outbound
Sends the personalized invite carrying each contact's unique voice link; reports opens and clicks.
ElevenLabs
interviewer
Runs the prescreen, hangs up on non-fits, returns transcript plus evaluation and extracted fields.
02

The flow, end to end

↓ = crm_id present

Read top to bottom as the happy path. The trace on the left is the join key — it stays live through every node, which is the whole design.

crm_id
01
HubSpotContact enters the target list

A filtered view (has email · persona mapped · tied to a probe company) fires a webhook, or a scheduled sync sweeps it. The HubSpot contact id becomes crm_id — the join key for the whole loop.

contact { crm_id, email, phone, company, persona, competitors }
02
SupabaseStage row, mint personalized link

Upsert into contacts. Build the per-person dynamic variables and a voice_survey_url — the hosted agent link with var_crm_id, var_vendor, var_persona baked in. Optionally a signed URL so it can't be forwarded or tampered with.

lead + {{voiceSurveyUrl}} custom var
03
LemlistSend the invite sequence

Push the contact into a campaign with the link as a custom variable. Framing: a two-minute voice check before we book a paid call. Sequence handles the reminder if unopened. The email is also where consent is gated, replacing the interviewer's built-in screen.

webhook opened / clicked
04
SupabaseTrack engagement

Lemlist activity updates outreach_state. A click is the intent signal — the contact is now walking into the call.

click → conversation starts, vars bound
05
ElevenLabsRun the prescreen

The hosted page opens the agent with the contact's variables already bound. Five questions, one probing follow-up each. The end_call tool hangs up early if they reveal they're not a fit — verified: it exits at turn 5 instead of grinding through all five.

Alt — phone

Skip Lemlist for contacts with numbers: Supabase submits a batch-calling job (one request, whole list, each recipient carrying its own variables) and ElevenLabs calls them. Same agent, same parsing — only the trigger differs.

post-call analysis (evaluation + data collection)
06
ElevenLabsScore and extract on completion

Once the call ends, an analysis LLM runs the evaluation criteria (tenure / depth / fraud → success·failure·unknown + rationale) and data-collection fields against the transcript. No separate judge pass needed.

HMAC webhook { conversation_id, transcript, analysis, crm_id }
07
SupabaseReceive, verify, flatten

An Edge Function verifies the signature, then keys on conversation_id and upserts (webhooks double-fire). It reads dynamic_variables.crm_id to find the row, flattens evaluation_criteria_results + data_collection_results into call_results, keeps the raw transcript.

PATCH properties { voice_* } on crm_id
08
HubSpotWrite back and route

Eight clean columns land on the contact. A workflow reads them: a genuine fit pings a researcher in Slack to book the paid call; a non-fit is marked disqualified with the reason the agent gave.

returns enriched
The loop closes.
The same row that left as name · email · persona · target company returns carrying a self-reported association and a credibility judgment on it — enough for a human to book or bin without listening to a single call.

04

Contact state machine

one row, one lifecycle

Every hop advances the state, so a stalled contact is always legible — you can see exactly where it fell out.

staged→ invited→ opened→ clicked→ in_call→ completed→ parsed→ enriched
routed_booktenure & depth pass, no fraud flag, follow-up worthy → researcher pinged to book the paid call.
disqualifiedagent invoked end_call, or tenure/depth failed → closed with the stated reason.
no_showclicked but never completed → drops back into the Lemlist reminder sequence.
fraud_flagintegrity criterion failed → held for human review, never auto-rejected.
05

Decisions that keep it real

Supabase is the spine, not HubSpot
Never stuff transcripts or nested analysis into the CRM. HubSpot gets flat final columns; Supabase owns the state machine, raw payloads, and idempotency.
crm_id threads every hop
Dynamic variable → conversation record → webhook → property write. It is the only thing that makes results joinable back to a person.
Idempotent webhooks
Post-call and Lemlist webhooks both double-fire. Key on conversation_id (or lead + event) and upsert — never append.
One agent, two triggers
Email link and phone batch share the same agent and parser. Persona and company are variables, not separate agents — no flow-per-persona sprawl.
Consent lives in the email
The invite and landing gate consent before the call, so there's one consent system feeding the existing compliance flow, not two.
Parsing is already done
ElevenLabs' analysis LLM fills the dimensions on completion. Add your own pass only beyond the 30-criteria / 40-field caps.
