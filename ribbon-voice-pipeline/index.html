System Design · Voice Prescreen Enrichment

The Voice Enrichment Loop

A closed loop that takes a CRM row, calls the person with an AI voice interviewer, and writes back what they actually claimed — and how much to believe it. Four systems, one join key, no human in the middle until a real expert is found.

The loop only closes because one value — crm_id — rides every hop: into the dynamic variables that personalize the call, back out on the conversation record, through the post-call webhook, and onto the HubSpot property write.
Everything else is plumbing. Get the join key right and the enrichment is deterministic; get it wrong and you have a pile of orphaned transcripts.
4systems in the loop
2delivery modes · email link / phone
8fields written back per contact
~$0.45all-in per 5-min call

01The four systems

Each does one job. The rule that keeps it sane: HubSpot stays flat, Supabase holds the mess. Transcripts, nested analysis JSON, webhook idempotency, and per-contact call state live in Supabase; HubSpot only ever sees clean, final columns.

HubSpot
System of record. Source of the contact list (email, phone, persona, target company) and destination for the enriched columns.
Supabase
The spine. Orchestration state, link/variable generation, webhook receiver, transcript + analysis store, parser.
Lemlist
Outbound. Sends the personalized invite sequence carrying each contact's unique voice-survey link; reports opens/clicks.
ElevenLabs
The interviewer. Runs the prescreen agent, hangs up on non-fits, and returns transcript + evaluation + extracted fields.
HubSpot→ Supabase→ Lemlist→ ElevenLabs→ Parse→ Supabase→ HubSpot  ⟲ enriched
Read left-to-right as the happy path; the last arrow is the whole point — the row returns to HubSpot fuller than it left.

02The flow, end to end

Colored left-edges show which system owns each step. Dashed boxes on the connectors are the payload passed forward.

Phase 1 Sync & stage
01HubSpotContact enters the target list
A filtered view (has email · persona mapped · associated with a probe company) fires a webhook — or a scheduled sync sweeps it. The HubSpot contact id becomes crm_id, the join key for the whole loop.
▼contact { crm_id, email, phone, company, persona, competitors }
02SupabaseStage row + mint personalized link
Upsert into contacts. Build the per-person dynamic variables and a voice_survey_url — the hosted agent link with var_crm_id, var_vendor, var_persona baked in. Optionally a signed URL so it can't be forwarded or tampered with.
Phase 2 Outbound
▼lead + {{voiceSurveyUrl}} custom var
03LemlistSend the invite sequence
Push the contact into a Lemlist campaign with the link as a custom variable. Email framing: "a 2-minute voice check before we book a paid call." Sequence handles the reminder if unopened. The email is also where consent is gated — replacing the interviewer's built-in consent screen.
▼webhook: opened / clicked
04SupabaseTrack engagement
Lemlist activity webhooks update outreach_state. A click is the intent signal — the contact is now walking into the call.
Phase 3 The call
▼click → conversation starts, vars bound
05ElevenLabsRun the prescreen
The hosted page opens the agent with the contact's variables already bound. Five questions, one probing follow-up each. The end_call tool hangs up early if they reveal they're not a fit (wrong person, no real exposure, declines) — proven: it exits at turn 5 instead of grinding through all five.
Alt path · phone
Skip Lemlist for contacts with phone numbers: Supabase submits a batch-calling job (one request, whole list, each recipient carrying its own dynamic variables) and ElevenLabs calls them. Same agent, same downstream parsing — only the trigger differs.
▼post-call analysis (evaluation + data collection)
06ElevenLabsScore & extract on completion
Once the call ends, an analysis LLM runs the configured evaluation criteria (tenure / depth / fraud → success·failure·unknown + rationale) and data-collection fields (tenure, role, strengths, weaknesses) against the transcript. No separate judge pass needed for these.
Phase 4 Parse & enrich
▼HMAC-signed webhook { conversation_id, transcript, analysis, crm_id }
07ParseReceive, verify, flatten
A Supabase Edge Function verifies the signature, then keys on conversation_id and upserts (webhooks double-fire). It reads dynamic_variables.crm_id to find the row, flattens evaluation_criteria_results + data_collection_results into call_results, and keeps the raw transcript + audio URL.
▼PATCH properties { voice_* } on crm_id
08HubSpotWrite back & route
Eight clean columns land on the contact (§03). A HubSpot workflow reads them: a genuine fit pings a researcher in Slack to book the paid call; a non-fit is marked disqualified with the reason the agent gave.
⟲
The loop closes.
The same row that left HubSpot as name · email · persona · target company comes back carrying a self-reported association and a credibility judgment on it — enough for a human to book or bin without listening to a single call.

04Contact state machine

One row, one lifecycle. Every hop advances the state, so a stalled contact is always legible — you can see exactly where it fell out.

staged→ invited→ opened→ clicked→ in_call→ completed→ parsed→ enriched
routed_book — tenure & depth pass, no fraud flag, follow-up worthy → researcher pinged to book the paid call.
disqualified — agent invoked end_call, or tenure/depth failed → closed with the stated reason.
no_show — clicked but never completed; drops back into the Lemlist reminder sequence.
fraud_flag — integrity criterion failed → held for human review, never auto-rejected.

05Decisions that keep it real

Supabase is the spine, not HubSpot
Never stuff transcripts or nested analysis into the CRM. HubSpot gets flat final columns; Supabase owns the state machine, raw payloads, and idempotency.
crm_id threads every hop
Dynamic variable → conversation record → webhook → property write. It is the only thing that makes results joinable back to a person.
Idempotent webhooks
Post-call and Lemlist webhooks both double-fire. Key on conversation_id (or lead+event) and upsert — never append.
One agent, two triggers
Email link and phone batch share the same agent and the same parser. Persona and company are variables, not separate agents — no flow-per-persona sprawl.
Consent lives in the email
The Lemlist invite + landing gate consent before the call, so there's one consent system feeding the existing compliance flow, not two.
Parsing is already done
ElevenLabs' analysis LLM fills the dimensions on completion. Add your own pass only for anything beyond the 30-criteria / 40-field caps.
