Tela · Tech design · 2026-07-15

A collaboration layer for persistent AI workspaces.

Each workspace is a live coding agent running in tmux on an always-on VM. Multiple humans attach to the same agent, watch the same bytes, steer it, and answer its approval prompts together. Closing the browser never stops anything. Live at tela-vm.web.app (Altis SSO).

03

Collaboration mechanics

  • One geometry. Every session is pinned 120×36 (window-size manual) — every participant sees byte-identical output; humans SSHing in can’t resize it out from under the room.
  • Attributed prompts. Messages go through the daemon: injected into the TUI (with a bracketed-paste-safe Enter) and broadcast as chat events with author + color.
  • Approvals for everyone. The daemon parses numbered agent dialogs from the screen (per-agent carets) and every participant gets clickable approve/deny buttons.
  • Permission modes. For Claude workspaces the active mode (Manual / Accept edits / Plan / Auto / Bypass) is parsed live and shown as a ⇧⇥ pill — click to cycle via shift+tab; everyone sees it change.
  • Presence, interrupts, unread. Join/leave toasts, header avatars, cross-window interrupt, unread-activity dots — all from real daemon events, nothing simulated.
  • Chat history. Prompts, interrupts, joins/leaves persist per workspace (ET timestamps) and replay to anyone who joins later — a livestream chat for the agent’s room.
  • Scrolling that actually works. Two paths, picked automatically: Codex/Gemini flow output into tmux history — 50k-line replay on join, local wheel. Claude Code/OpenCode repaint an internal transcript that never touches the terminal, so the wheel is forwarded to the agent as synthetic mouse events and it scrolls its own transcript — live for every participant.
  • Workspaces from the web. Any signed-in user creates one: name + agent + repo path on the server (allowlisted roots); first-run dialogs are answerable from the approval bar.
04

Trust model

The app is public static files; everything sensitive sits behind the daemon. Any Altis SSO user can connect to any Tela server that trusts Research OS; non-SSO collaborators use that server’s access code. Participants are collaborators by design — connected users have full agent control. Agent API keys live only on the VM.

Known limits: the VM endpoint rides on an ephemeral IP (a stop/start means updating one JSON file + redeploying hosting); chat history travels with the server, not the user; dynamic terminal resize is deliberately out of scope.
